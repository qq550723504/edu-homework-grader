# 隐私与审计技术基线设计

## 目标

在试点前为未成年人教育数据建立可执行的最小化收集、去标识化、日志脱敏和防篡改审计基线。本设计只覆盖 Issue #8 的第一个交付切片；监护人同意、数据主体请求、备份恢复演练和 PIA 另行实施。

## 范围与边界

本切片覆盖：

- 学生身份字段、OIDC claim 和业务数据之间的最小化边界。
- API 到批改服务、以及未来外部处理器的去标识化数据契约。
- 应用日志的敏感字段拒绝与结构化安全事件。
- 登录、导出、规则变更、成绩发布和教师改分的可验证审计记录。
- 数据清单和各类数据的默认保存/删除规则。

本切片不实现：

- 监护人身份校验、同意采集或撤回界面。
- 查阅、更正、删除和导出申请工单。
- 跨系统备份恢复、删除作业或安全事件演练。
- 法律意见、PIA 审批或对外模型供应商采购。

## 当前基线

- `users.id` 已是内部 UUID；`student_attempts.student_id`、作答和评分记录只关联该 UUID。
- 学生记录受 `ck_students_require_school_id_and_no_email` 约束，不能保存工作邮箱；当前 schema 不含手机号字段。
- 首期批改 HTTP 请求仅包含题目规则与答案，不含学生 UUID、姓名、学校编号或 OIDC subject。
- English E3 的 LanguageTool 和 E4 的语义模型是试点 Compose 内依赖；它们仍被视为数据处理器，不能因当前部署位置而绕过最小化约束。
- 当前 `audit_logs` 是可写业务表，`metadata_json` 没有限制，且尚未覆盖所有受控事件；它不能单独证明未被事后篡改。
- Keycloak Web 客户端默认请求 `profile` 与 `email` scope。即使学生邮箱为空，该能力也超出本切片的身份最小化需求。

## 方案比较

### 方案 A：仅维护合规文档

成本低，但数据和日志仍可在运行时越界，无法作为试点的技术控制。

### 方案 B：文档加运行时强制控制（采用）

保留现有 UUID 业务关联，收紧 OIDC scope，定义处理器数据契约，集中清洗日志，并以哈希链和服务端 HMAC 签名提供篡改可检测的审计账本。它可独立测试、可在 Compose 环境实施，并为后续流程提供稳定基础。

### 方案 C：一次完成完整合规项目

还要决定监护人验证方式、法定保存期限、数据请求 SLA、备份供应商和 PIA 审批人。它跨越产品、运维和法务决策，不应阻塞可验证的技术基线。

## 设计

### 1. 数据分类与保存规则

在 `docs/data-inventory.md` 中维护代码拥有的数据清单。每一项必须包含字段/记录类型、用途、敏感等级、允许访问角色、处理器、默认保存期限、删除触发条件及关联的数据库表。

分类使用 `restricted`（未满十四周岁学生资料、身份绑定、作答正文、成绩、申诉理由）、`confidential`（学校编号、教师邮箱、班级归属、审计关联标识）和 `internal`（非个人化配置与聚合指标）。任何新 schema 字段或外部处理器都必须先更新清单。

本切片为工程默认值，不取代学校或法务确定的法定保存期限：草稿作答在提交后 30 天删除；已发布作答、评分和申诉记录保存 2 年；审计账本保存 3 年；无法删除的记录进入 `erasure_pending` 状态，仅保留和执行必要保护。具体租户上线前可由受控配置收紧，但不得延长。

### 2. 身份最小化与处理器契约

学生的唯一业务关联继续使用 `users.id`。`school_id` 只用于可信花名册与首次 OIDC 绑定，不能传给作答、评分、复核或审计之外的服务。学生手机号不加入用户、OIDC 或前端 schema。

Keycloak Web 客户端不再默认请求 `email` 或 `profile` scope；API 只接受完成认证和花名册绑定所需的 `sub`、`iss` 与 `school_id`。显示名称由本地 UUID 用户记录提供，不作为批改请求字段。

引入一个处理器出站数据契约：`GraderPayload` 只能含题型、策略版本、题目规则和答案；禁止 identity 字段、班级、租户 slug、时间戳和自由格式 metadata。外部处理器使用显式配置 allowlist；未列入 allowlist 的 URL 在生产环境启动失败。每个处理器在清单中声明部署位置、用途、字段和保留策略。

### 3. 安全日志

应用通过唯一的结构化日志适配器输出运行日志。适配器递归拒绝或掩码以下键和值：`answer`、`answer_json`、`authorization`、`token`、`password`、`oidc_subject`、`school_id`、姓名、邮箱和连接字符串。日志仅记录事件代码、请求/关联 UUID、HTTP 状态、耗时和受控的错误类别。

禁止业务代码直接对用户输入、请求头、ORM 对象或异常对象调用日志方法。测试将构造答案、令牌和身份字段，证明日志输出不包含原文；同时验证换行和控制字符不会造成日志注入。

### 4. 防篡改审计账本

审计记录沿用业务数据库以便按租户检索，但新增以下字段：单调序号、前一条记录哈希、当前记录哈希、HMAC 签名、密钥版本和不可变的规范化事件载荷。哈希覆盖租户、执行者 UUID、事件类型、目标、发生时间、规范化 metadata、序号和前一哈希；签名密钥来自部署 secret，绝不写入数据库或日志。

每个租户的序号和链头在同一事务中锁定并追加。数据库 migration 创建触发器，拒绝应用角色对审计表的 `UPDATE` 与 `DELETE`；只有专用维护角色可在受控恢复流程中操作。校验服务可从任意起点重算链和 HMAC，并报告第一条异常记录。

这提供的是技术上的篡改可检测性和服务端签名证据；在法律意义上实现不可抵赖仍需要学校身份治理、OIDC 事件保留和独立的只写/外部时间戳存储。试点上线前必须将每日日志链头导出到独立、权限隔离的存储。

受控事件固定为：`auth.login_succeeded`、`auth.login_denied`、`data.export_requested`、`data.export_completed`、`question.rule_changed`、`grade.published`、`review.score_adjusted`。事件 metadata 只允许动作原因、旧/新版本号、记录计数、结果和关联 UUID，不能含答案、姓名、学校编号、令牌或完整请求体。

Keycloak 登录事件由 realm 的事件保留与导出承担；API 在已验证身份建立会话时记录成功绑定和访问拒绝。两者使用同一 correlation ID，并在数据清单中分别标明保留位置和访问角色。

### 5. 错误处理与运行行为

审计写入、签名或链校验失败时，受控写操作必须回滚并返回通用错误，不能继续发布成绩或变更规则。普通读取不因审计存储短暂故障泄露详情。生产环境若未配置审计 HMAC key、处理器 allowlist 或日志安全配置，服务拒绝启动。

## 测试策略

- 单元测试覆盖 payload 构造，断言每个禁止的身份字段均无法进入批改或外部处理器请求。
- 配置测试覆盖 OIDC scope 和生产启动时所需的安全配置。
- 日志测试覆盖秘密与学生答案脱敏、嵌套对象和日志注入字符。
- 审计测试覆盖每种受控业务事件、同租户并发追加、跨租户链隔离、篡改检测，以及数据库拒绝更新和删除。
- API 集成测试覆盖审计失败时发布成绩、改分和规则变更被原子回滚。

## 外部依据

- 《个人信息保护法》将不满十四周岁未成年人的个人信息列为敏感个人信息，并要求取得监护人同意和制定专门处理规则；后续同意切片据此设计。
- OWASP Logging Cheat Sheet 建议不直接记录访问令牌和敏感个人信息，并要求保护日志免遭未授权访问、修改和删除；本切片采用集中脱敏与篡改可检测账本落实该原则。
