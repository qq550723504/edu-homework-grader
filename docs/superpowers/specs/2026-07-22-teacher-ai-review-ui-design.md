# 教师 AI 候选题审核界面设计

**日期：** 2026-07-22

**范围：** GitHub #41 的教师 Web 审核切片。它消费已合并的候选题审核 API，让教师浏览已生成批次、查看验证证据、编辑、拒绝和接受单题。新建生成请求、单题重生成与批量接受留给下一切片。

## 根因与边界

后端已经以不可变修订、验证运行和题库草稿桥接保证审核安全，但教师端没有入口消费这些契约，审核 API 因而无法形成教学工作流。现有 `/teacher` 页面已承载概览、题库、作业和花名册；把另一套列表/详情/编辑状态嵌入其中会继续扩大单页复杂度。

界面只访问 Nuxt 的同源 `/api/core/v1/*` 代理。浏览器不保存或展示 Provider 密钥、系统 Prompt、私有教师约束、请求摘要或原始验证内部特征；页面只使用后端已经净化过的候选和验证投影。后端继续是授权、状态转换、版本并发与幂等的唯一权威。

## 已评估方案

1. **在 `/teacher` 的 hash 模块中嵌入审核区。** 复用现有页面，但会把已经很大的教师工作台继续耦合；不采用。
2. **新增独立的 `/teacher/ai-questions` 页面（采用）。** 批次列表、候选详情和编辑状态都有可分享/可恢复 URL，且与现有“复核队列”路由一致。
3. **只在题库页面使用弹窗。** 难以并列展示验证证据、编辑表单和拒绝原因，也不能恢复选中批次；不采用。

## 页面和数据流

导航新增“AI 出题审核”，指向 `/teacher/ai-questions`。页面读取 `job` 与 `draft` 查询参数：未指定批次时自动选择最新可访问批次；未指定候选时选择该批次第一道候选。切换批次或候选仅更新查询参数，刷新页面后由 API 重新加载真实服务端状态，不把审核状态持久化在浏览器。

```text
GET jobs -> 选择 job -> GET job questions -> 选择 draft
                                        |
                                        +-> POST revision -> 返回 validation_run -> 刷新候选
                                        +-> POST reject   -> 返回 decision       -> 刷新候选
                                        +-> POST accept   -> 返回 QuestionVersion -> 刷新候选
```

`apps/web/app/lib/teacher-ai-review.ts` 集中定义 API 投影类型和请求函数：列出任务、列出候选、创建修订、拒绝、接受。所有写请求从当前会话取得 CSRF token，并以 `crypto.randomUUID()` 生成一次操作一个的 `Idempotency-Key`；重试同一用户动作使用同一个进行中的 key，成功或失败后才允许新操作。错误映射只显示稳定的公共错误码和中文操作提示。

页面组件按职责拆分：

- `TeacherAiReviewWorkspace.vue` 负责加载、URL 同步、写操作、刷新和跨组件错误状态；
- `TeacherAiJobList.vue` 只渲染任务状态、请求数、成功/失败数和失败码；
- `TeacherAiCandidateReview.vue` 渲染一个候选的内容、编辑、验证证据与审核动作；
- `teacher-ai-review.ts` 不包含 Vue 状态，只包含可单测的请求契约与小型映射函数。

在窄屏下列表位于详情上方，候选操作始终保持可访问的全宽按钮；大屏采用两列 Grid。不会在组件内使用固定像素宽度导致横向滚动。

## 候选展示与编辑

详情显示顺序为：题型/修订/审核状态，E4 阅读材料，题干，答案规则，解析，知识点与难度，最后是验证证据。`rule_json` 显示为可读 JSON，并提供可编辑文本域；编辑表单以当前候选完整副本作为提交基础，允许更改：`prompt`、`rule_json`、`explanation`、`knowledge_point`、`difficulty` 和 E4 `reading_material`。`objective_revision_id`、`question_type`、`policy_version` 只读且不会由 UI 提交时改写。

保存前在客户端检查 JSON 是否可解析、难度是否在 0–1、E4 材料非空且非 E4 不携带材料。客户端检查只改善即时反馈；服务端 `GeneratedCandidate` 严格校验仍决定结果。保存调用 revisions API，后端同步运行验证；界面用返回的运行替换可视证据并重新读取候选，因此编辑标准答案会确实触发新验证而不是只更新本地预览。

## 验证与审核动作

验证区按运行状态展示 passed、warning、blocked，逐条展示公共 `code`、`severity`、`evidence` 和 `remediation`。无验证运行、加载中和请求失败各有明确空状态；不把缺失证据伪装为通过。

- `blocked`：接受按钮禁用，并说明必须编辑和重新验证。
- `warning`：接受前出现确认复选框；未确认时按钮禁用。
- `passed`：可直接接受。
- `pending_review`：可编辑、拒绝或接受；`accepted`/`rejected` 仅展示已完成状态，所有写按钮禁用。
- 拒绝打开内联面板，原因限定为后端枚举；选择 `other` 时要求 1–500 字说明。
- 接受成功显示“已创建题库草稿”及 `QuestionVersion` 标识和前往现有题库工作台的链接；不会发布，也不会绕过测试运行。

冲突、配额、Provider 不可用、授权隐藏的 404 与网络错误均保留当前服务端数据、解除按钮 loading 并给出可重试提示。对 `review_revision_conflict`，页面重新加载该候选并提示教师最新修订已显示。

## 明确不做

本切片不添加生成请求表单、课程 profile 级联、成本估算、任务轮询、单题重生成、批量接受或 Playwright 的“生成请求”链路。这些功能需要单独对生成配置、配额和批量幂等进行设计；在本切片中，任务列表只展示已存在批次。

## 测试与验收

- Vitest：请求函数使用正确 URL、CSRF 和幂等头；候选编辑保留锁定字段；JSON/E4/难度检查；状态到按钮可用性的映射；warning 必须确认，blocked 永不接受。
- DOM 测试：渲染 E4 材料、验证证据、拒绝原因输入和接受后的题库草稿提示；验证编辑后的返回运行替换原证据。
- Playwright：使用测试 Provider 已产生的候选，教师打开批次、编辑标准答案、观察新验证、拒绝另一题并接受通过题，确认题目只作为 draft 出现在现有题库流程。
- 所有现有 Web 测试和 Nuxt 构建继续通过；页面不包含任何 Provider 密钥、Prompt 或私有验证字段。
