# Tenant, Identity, and Authorization Design

## Goal

Implement Issue #2 as the security foundation for the first pilot: tenant-isolated users, classes, roles, roster import, and audit logging. The slice must be usable by later question, assignment, submission, and review work without introducing a local password store or leaking a record's existence across tenants.

## Scope

This design covers only Issue #2. It does not implement questions, assignments, submissions, grades, review queues, or a production school-SSO integration.

## Selected approach

Use Keycloak as the development OpenID Connect provider. The Nuxt client will later use Authorization Code with PKCE. The API is a resource server: it accepts Bearer access tokens and verifies them against the configured issuer's OIDC Discovery document and JWKS. It validates the token signature, algorithm, issuer, audience, expiry, and subject before using any identity claim.

This uses mature components rather than creating login, password, or token-signing code in the product:

- Keycloak for OIDC users, credentials, login flows, and development roles.
- SQLAlchemy 2.0 for relational persistence.
- Alembic for PostgreSQL schema migrations.
- PyJWT with the `crypto` extra for JWKS-based JWT verification.

Keycloak's development container imports a versioned `edu-grader-realm.json` from `/opt/keycloak/data/import` at startup. The imported realm provides the API audience, the three pilot roles, a stable `school_id` token claim, and test users. Production deployment replaces the development issuer through configuration; the API depends on OIDC Discovery, not Keycloak-specific endpoints.

## Data model

All identifiers below are UUIDs generated by the application except the external OIDC subject.

### `tenants`

`id`, `slug`, `name`, `created_at`. `slug` is globally unique and is used only for administrator configuration and test fixtures, never selected by an untrusted request header.

### `users`

`id`, `tenant_id`, `oidc_issuer`, `oidc_subject`, `role`, `school_id`, `display_name`, `work_email`, `created_at`, `updated_at`.

- `role` is exactly one of `admin`, `teacher`, or `student` for the pilot.
- Student rows require `school_id`, must have `work_email = NULL`, and are unique on `(tenant_id, school_id)`.
- Teacher and administrator work emails are optional and may not be used as a login identifier.
- `(oidc_issuer, oidc_subject)` is unique when bound. It is nullable before a rostered student logs in for the first time.
- The verified `school_id` claim binds a first-login student to the imported row inside the tenant. A bound row must match the same issuer and subject on every later request; a different subject is rejected.

The initial pilot administrator is explicit configuration, not silent just-in-time provisioning. `BOOTSTRAP_ADMIN_SUB` and `BOOTSTRAP_ADMIN_TENANT_SLUG` select the single known development administrator; the bootstrap command creates its row once. All other unknown identities receive 403 until an administrator imports or creates their membership.

### `classes`

`id`, `tenant_id`, `code`, `name`, `created_at`, `updated_at`. `(tenant_id, code)` is unique.

### `class_teachers`

`class_id`, `teacher_id`, `created_at`, with `(class_id, teacher_id)` as the primary key. `teacher_id` must reference a tenant-local user with role `teacher`.

### `enrollments`

`class_id`, `student_id`, `created_at`, with `(class_id, student_id)` as the primary key. `student_id` must reference a tenant-local user with role `student`.

Database constraints and service checks enforce that every linked user belongs to the same tenant as the class. Future business tables use a shared SQLAlchemy tenant mixin that supplies non-null `tenant_id` and a foreign key to `tenants`.

### `audit_logs`

`id`, `tenant_id`, `actor_user_id`, `event_type`, `target_type`, `target_id`, `occurred_at`, `metadata_json`. Rows are append-only; there is no update or delete service. Metadata contains identifiers and changed field names only, never answers, access tokens, full JWT claims, or passwords.

## Authorization and request flow

1. The API dependency reads the Bearer token and validates it using the configured issuer, audience, Discovery metadata, and JWKS key matching `kid`.
2. It finds the user by verified `(issuer, sub)`. The trusted server-side `OIDC_TENANT_SLUG` configuration selects the only tenant eligible for the issuer in this pilot. If the identity is an unbound rostered student, it matches the verified `school_id` claim only inside that tenant and binds the subject transactionally. It never creates an ordinary user from an arbitrary token.
3. The request receives a `CurrentPrincipal` containing the internal user UUID, tenant UUID, and database role. Route dependencies use this value, not a client-supplied tenant ID or raw token role.
4. Admin routes require `admin`. Teacher class routes require an assignment in `class_teachers`. Student class routes require an `enrollments` row for the current student. All entity queries include the principal's `tenant_id`.

An unauthenticated or invalid token returns 401. An authenticated identity that is unknown, has the wrong role, belongs to another tenant, or is not enrolled or assigned to the requested class returns 404. The 404 response prevents tenant and membership enumeration.

## API contract

- `GET /v1/me` returns `{id, tenant_id, role, school_id, display_name}` for the current principal. It returns 403 for a verified token without an authorized platform membership.
- `POST /v1/admin/students/import` accepts a CSV upload with `class_code,class_name,student_school_id,student_display_name`. The endpoint creates or updates classes, students, and enrollments in one transaction. Existing enrollment rows are retained; no role is changed by import.
- `POST /v1/admin/classes` creates a tenant-local class.
- `PUT /v1/admin/classes/{class_id}/teachers/{teacher_id}` assigns an existing tenant-local teacher to a class.
- `GET /v1/classes/{class_id}` proves class authorization and returns its minimal identity metadata. Assignment and question payloads are deliberately deferred.
- `GET /v1/admin/audit-logs` returns tenant-local audit entries with pagination and optional event-type/time filters.

Malformed CSV, duplicate student IDs in the same input, unknown referenced users, and invalid class codes return 422 with the source row and field. The import performs no partial write. Reimporting the same roster updates the imported student display name and preserves existing enrollment relationships.

## Configuration and operations

Add `DATABASE_URL`, `OIDC_ISSUER`, `OIDC_AUDIENCE`, `OIDC_SCHOOL_ID_CLAIM`, `OIDC_TENANT_SLUG`, `BOOTSTRAP_ADMIN_SUB`, and `BOOTSTRAP_ADMIN_TENANT_SLUG` to application settings and `.env.example`. Use a single PostgreSQL Compose service for the pilot database; Keycloak has its own PostgreSQL database/schema configuration so identity data remains outside the application tables. Compose starts Keycloak before services that need its Discovery endpoint, but API tests inject a verifier and never depend on the network or a running identity provider.

Alembic is the only supported schema change mechanism. Application startup does not call `create_all`; local Docker startup runs `alembic upgrade head` before serving requests.

## Tests and acceptance mapping

Tests are written before implementation and exercise real route dependencies and database transactions. They cover:

1. Invalid signature, issuer, audience, expiry, and missing subject each return 401; a valid injected verifier produces a principal.
2. A rostered student's first verified login binds the OIDC subject once; a later different subject cannot take over the row.
3. Tenant boundaries reject cross-tenant class, user, roster, and audit access with 404.
4. A teacher can read only an assigned class; a student can read only an enrolled class; an administrator is limited to their own tenant.
5. CSV validation failures roll back every intended insert; a repeated valid import is idempotent; successful changes append audit events.
6. A fresh PostgreSQL database migrates to the Alembic head with the required primary keys, tenant foreign keys, unique constraints, and audit table.

The slice satisfies Issue #2's tenant/user/class/enrollment models, three roles, teacher/student boundaries, student/class roster import interface, cross-tenant tests, and audit requirements. It intentionally defers product and grading data to dependent issues.
