# Epic #1 端到端验收设计

## 目标

为 Issue #1 的教师建题到学生订正主链路建立两层可重复验收：一条 API 垂直测试覆盖所有业务状态转换；一条 Playwright 浏览器测试覆盖学生端与真实 API 的交互。浏览器测试使用专用测试身份，不接入 Keycloak，也不能改变生产认证行为。

## 范围

API 验收测试以一个租户、一个班级、一个教师和一个学生为场景，按以下顺序执行：

1. 教师创建数学表达式题及其草稿版本。
2. 教师配置标准答案、变量、总分、答案形式和正确、错误、空值、边界测试用例；运行用例后发布版本。
3. 教师创建作业、加入已发布题目并发布给班级。
4. 学生打开作业，保存 MathJSON 答案，并使用幂等键提交。
5. 测试断言评分运行保存题目版本、规则版本、批改器版本和代数等价证据。
6. 教师读取评分证据、确认结果并发布成绩。
7. 学生读取已发布反馈、提交申诉；教师批准后产生独立订正尝试。学生提交订正答案，教师再次发布结果。

每个步骤必须经过对应 HTTP 路由，而不是直接构造领域模型；断言只读取该角色可见的投影。该测试不覆盖支付、OCR、作文自动最终评分或复杂几何证明。

## 浏览器验收边界

浏览器测试不重做教师后台 UI（该 UI 当前不存在）。它通过 API 测试准备已发布作业，再启动 Nuxt 与专用 E2E API 应用，验证学生端可以：

1. 使用短期测试令牌加载作业列表和详情；
2. 输入并同步答案；
3. 提交作业并看到已提交状态；
4. 在 API 发布结果后看到学生安全反馈和订正入口。

使用 Playwright，因为它是成熟的跨浏览器端到端测试框架，能够同时管理 Nuxt 服务、浏览器上下文和失败追踪资料。测试固定使用 Chromium，避免将试点验收变成多浏览器兼容性项目。

## 测试认证和服务隔离

新增独立的 `edu_grader_api.e2e_app` ASGI 入口，仅被 E2E 命令启动。该入口在进程内覆盖令牌验证器和批改器依赖：固定 bearer token 映射到已播种的教师或学生身份，批改器返回确定性 M2 结果。

生产 `edu_grader_api.main:app` 不导入、不注册、也不接受这些令牌；生产 OIDC 设置和 Keycloak 流程保持不变。E2E 入口仅绑定回环地址，并且命令为每次运行创建独立临时 SQLite 数据库。迁移、播种、Nuxt、API 和浏览器进程的启动与停止均由测试脚本管理；测试完成后删除数据库和临时资料。

## 命令与质量门槛

在 Web 包中新增 `npm run test:e2e`，负责启动 Playwright 测试。首次使用时通过 Playwright 官方命令安装 Chromium。现有 `npm test`、API pytest 和生产 Compose 启动方式不改变。

验收通过条件：API 测试可以从空测试库重复执行；Playwright 在真实 Nuxt 页面与真实专用 API 间完成学生交互；测试令牌不能被生产 ASGI 应用接受；所有既有 Python 与 Web 测试继续通过。

## 错误处理和可观测性

任何 API、Nuxt 或浏览器启动失败都应使 E2E 命令失败，而非跳过。Playwright 保留失败截图、trace 和浏览器控制台输出；这些资料不应包含真实学生数据，因为播种身份和答案均为固定虚构数据。
